Processes

A container is responsible for starting one process, and once that process exits, the container exits.

  • Container will be in the exit state, it will not be removed and still exists on disk (no data loss)

 

There is no init system that starts up multiple services like you are traditionally used to on VMs

 

When you require multiple processes in a container, you really should ask yourself why?

  • Is the container serving multiple roles / jobs?  Split them up into separate containers
  • Dockerizing legacy applications sometimes require it 

Processes

To have multiple processes running in the container

  • Chain commands
    • Nice for quick solution but not usually suitable longterm 
  • Use supervisord or similar service management process
    • Preferred way and allows future flexibility

Ports and Linking

Ports inside the container can be punched through the hosts

  • EXPOSE keyword in Dockerfile
  • --publish or --expose argument for docker run

Linking will inject environment variables and update the /etc/hosts of the container

  • https://docs.docker.com/userguide/dockerlinks/​

Allows containers to dynamically find each other

  • Should not rely on static port mappings, that forces constraints on what can run on the host

 

Volumes

Data for containers can be managed by using volumes

  • https://docs.docker.com/userguide/dockervolumes/​

Volumes are basically bind mounts to directories on the host machine

  • Persists even if container is deleted
  • Useful for sharing resources 
    • Access datasets available on the host
    • Mount the source of a project to build inside a container

Container can be made to provide access to a volume

  • Data Volume Container
  • Containers can inherit the volumes using the --volumes-from argument to docker run

Ansible

Using Ansible or your flavor of provisioning tool also keeps the Dockerfile clean of application logic

FROM infinidb:base
MAINTAINER Michael Hoglan <michaelh@tune.com>

# Provide the playbook vars for this image
COPY playbook_vars.yml /opt/tune/playbooks/vars/

WORKDIR /opt/tune/playbooks/

# Perform InfiniDB installation
# Cleanup yum which playbook may have used
# Stop the service (installation auto starts it) 
RUN ansible-playbook -i localhost_launched docker_infinidb.yml -e "@vars/playbook_vars.yml" --connection=local && service infinidb stop && yum clean all

EXPOSE 3306
CMD ["/usr/bin/supervisord", "--configuration", "/etc/supervisord_infinidb.conf"]

Supervisord

InfiniDB uses an init.d script for service actions

  • Most init.d scripts are not blocking and are not signal aware.

Instead of using the service script and chaining a blocking command to the docker run, use supervisord

  • http://supervisord.org/
  • https://docs.docker.com/articles/using_supervisord/

Existing scripts can be wrapped in a bash script which is blocking and signal aware

  • Allows using the service with supervisord without refactoring
  • https://github.com/TuneOSS/InfiniDB-Tools/blob/master/playbooks/roles/infinidb_supervisor/templates/infinidb.sh.j2

Building with Docker

Changing your build environment to be container based is one of the quickest ways to get into Docker

  • Build servers become generic resource pools

No longer need to keep separate build servers with different environments to build specific projects

  • Build needs are encapsulated in the container
  • Pushes the responsibility to the developer
    • They are the best resource for the application
    • Promotes being a full stack engineer

DevOps and SCM focus on their primary tasks, providing infrastructure to publish products and supporting production environments.

 

Building with Docker

Utilize a Makefile or any other build script utility to encapsulate the docker commands 

make build release rpm

Example of building a golang project inside a Docker container and copying the resultant artifact to the host

docker run --rm --entrypoint /bin/sh \ 
   -v /root/telemetry-go:/go/src/github.com/TuneDB/telemetry-go \ 
   -v /root/telemetry-go/target:/export \
   -w /go/src/github.com/TuneDB/telemetry-go golang:latest \
   -c "make go-install && cp /go/bin/receiver /export && make build-tar"

Recursively uses the Makefile of the project to execute inside the Docker container

  • Allows developers to use the other Makefile targets directly

Packaging into Docker Image

Artifacts of builds can also be packaged up into a Docker image to be deployed.

  • Works great when the artifact is an executable (such as a JAR or golang binary) that can be the long running process of a container

Package artifacts into minimal containers such as the scratch container

  • Keeps the resultant image size small
  • Useful for when shipping containers to different hosts that may not have the base image layer used by the container

Minimal Docker Release Images

Based our work on the thoughts from this blog post

  • http://blog.charmes.net/2014/11/release-go-code-and-others-via-docker.html

 

Our Makefile has a build-tar target which packages up the binary and /etc/ssl from the build container into a tar

  • /etc/ssl is needed for Linux TLS, easiest to bring with binary

 

Dockerfile for the release image

FROM scratch
MAINTAINER Michael Hoglan michaelh@tune.com

ADD ./target/receiver.tar /

ENTRYPOINT ["/go/bin/receiver"]
